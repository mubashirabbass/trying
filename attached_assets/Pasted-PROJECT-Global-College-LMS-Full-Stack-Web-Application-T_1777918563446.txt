VIDEO SYSTEM (YouTube Unlisted — Full Protection):

CRITICAL REQUIREMENTS:
1. Never expose the YouTube video ID directly in the HTML source, API response,
   or browser network tab
2. Store YouTube video IDs encrypted in the database using AES-256 encryption
3. Create a backend endpoint: GET /api/student/lessons/:lessonId/stream-token
   - Verifies the student is enrolled in the course this lesson belongs to
   - Verifies the student's session is valid
   - Decrypts the YouTube video ID server-side
   - Returns a short-lived signed token (expires in 30 minutes)
4. Frontend uses this token to fetch the embed URL — never the raw YouTube ID
5. Render the video inside a sandboxed iframe:
   - sandbox="allow-scripts allow-same-origin"
   - No YouTube branding if possible (use youtube-nocookie.com domain)
   - Add: modestbranding=1, rel=0, controls=1, disablekb=0
   - Block right-click on the video iframe using a transparent overlay div
   - Add CSS: pointer-events: none on the iframe, pointer-events: all on the overlay
   - The overlay div catches right-click and shows a custom "Content Protected" message
6. Do NOT allow the video URL to be opened in a new tab
7. Log every video access attempt (userId, lessonId, timestamp, ipAddress)
   in a VideoAccessLog table

VIDEO COMPLETION TRACKING (100% accurate):
1. Inject a YouTube IFrame API listener in the video component
2. Poll the player's getCurrentTime() and getDuration() every 5 seconds
3. Calculate watchedPercent = (currentTime / duration) * 100
4. Every 10 seconds, call PATCH /api/student/lessons/:id/progress with:
   { watchedPercent, currentPositionSeconds }
5. Backend upserts LessonCompletion record with latest watchedPercent
6. When watchedPercent reaches completionThreshold (default 80%, configurable
   in Admin Settings), mark lesson as complete and trigger course completion check
7. On reload, fetch the student's lastPositionSeconds and resume from there
8. Use YouTube's onStateChange event to detect:
   - PLAYING: start tracking interval
   - PAUSED / ENDED: stop tracking interval, save progress immediately
9. Prevent completion fraud: if video duration is < 60 seconds and student
   jumps directly to end position, require actual elapsed watch time

BACKEND ARCHITECTURE (Node.js + Express + Prisma):

Folder structure:
/server
  /src
    /controllers    — one file per domain (auth, courses, payments, etc.)
    /routes         — one file per domain, import controllers
    /middleware     — auth.js, rbac.js, upload.js, validate.js, rateLimiter.js
    /services       — email.js, notification.js, certificate.js, video.js,
                      report.js, completion.js, encryption.js
    /utils          — response.js (standard API response), logger.js, errors.js
    /validators     — input validation schemas using express-validator
  /prisma
    schema.prisma
    /migrations
  app.js            — express app setup
  server.js         — starts the server
  .env

Standard API response format (always use this):
Success: { success: true, data: {...}, message: "..." }
Error:   { success: false, error: "...", code: "ERROR_CODE" }

All controllers use async/await wrapped in a tryCatch utility.
All routes use express-validator for input validation.
All authenticated routes use authenticate middleware (verify JWT).
All role-restricted routes use authorize("ADMIN") or authorize("TEACHER","ADMIN").

SECURITY REQUIREMENTS:
- helmet() on all routes
- cors() configured for frontend domain only (from env variable)
- express-rate-limit: 5 requests per 15 minutes on /api/auth/login
- express-rate-limit: 10 requests per hour on /api/auth/register
- bcrypt password hashing with cost factor 12
- Account lock after 5 failed logins — locked for 15 minutes
- All file uploads: validate MIME type server-side before accepting
- YouTube video IDs stored AES-256 encrypted in the database
- Never log passwords, tokens, or video IDs to console
- Environment variables for all secrets — nothing hardcoded
- Admin route base path should be /api/admin (not guessable path)
- CSRF protection on state-changing endpoints
- Input sanitization on all text fields (xss-clean)

ENROLLMENT RULES:
- Free course: POST /api/enrollments → immediate enrollment, no payment needed
- Paid course: student submits payment receipt → admin manually approves
  → enrollment created only after approval → never before
- Admin can manually enroll any student in any course bypassing payment
- One enrollment record per student per course (unique constraint)

- Use Prisma transactions for any operation that writes to multiple tables
- All database queries go through the Prisma client — no raw SQL except for reports

- Use Prisma's select to never return password, encrypted video IDs, or sensitive fields
- API versioning: all routes under /api/v1/
- Write a README.md with setup instructions, env variable documentation, and API overview
